# 设计文档

本文档面向维护者，说明当前项目为什么这样设计、请求在内部如何流转，以及哪些地方是刻意保持轻量的。

## 1. 目标与边界

当前项目的目标：

- 用 FastAPI 提供标准 RESTful API
- 聚合外部 GitHub 和天气数据
- 具备最基本的公网暴露安全措施
- 让团队在一致的 Python / Docker 环境里开发和交付

当前项目刻意不做的事情：

- 不引入 Alembic
- 不引入 JWT / OAuth
- 不接入复杂权限系统
- 不拆分成微服务
- 不引入异步任务队列

这些都不是不能做，而是当前业务规模还不需要。

## 2. 目录结构

```text
app/
├─ api/          # 路由与 OpenAPI 示例
├─ core/         # 配置、异常、数据库、缓存、日志、安全
├─ models/       # SQLModel 模型
├─ schemas/      # Pydantic 输入输出模型
├─ services/     # 业务逻辑与外部 API 调用
└─ main.py       # 应用启动入口
```

职责划分原则：

- `api` 只负责路由定义、参数声明、响应声明
- `services` 只负责业务流程和外部依赖调用
- `core` 只放横切能力
- `schemas` 只定义返回结构
- `models` 只定义持久化结构

## 3. 请求流转

### 3.1 业务接口请求流

1. 请求进入 FastAPI
2. 中间件生成或透传 `X-Request-ID`
3. 路由层执行 API-Key 校验
4. 路由层调用 service
5. service 决定读缓存、读数据库、还是请求外部 API
6. 结果被封装成统一成功响应
7. 出错时交给统一异常处理器转换成标准错误响应

### 3.2 失败请求流

1. 任意层抛出 `ApiException` / `UpstreamServiceError` / 框架异常
2. 统一异常处理器生成 `code + message + details + request_id`
3. 响应头继续保留 `X-Request-ID`

## 4. 配置管理

项目使用 `pydantic-settings` 读取配置。

关键配置项：

- `API_KEY`
- `REDIS_URL`
- `SQLITE_URL`
- `GITHUB_CACHE_TTL_SECONDS`
- `WEATHER_CACHE_TTL_SECONDS`
- `APP_ENV`
- `LOG_LEVEL`

配置来源优先级由运行环境控制，默认通过 `.env` 注入。

## 5. 鉴权设计

当前只实现了一个简单但足够实用的公网防护方案：

- 通过 `X-API-Key` 做静态密钥校验
- 健康检查接口不强制认证
- 业务接口统一挂在路由级依赖上

为什么先这样做：

- 成本低
- 前后端接入简单
- 比匿名暴露到公网安全得多
- 当前阶段不需要为 JWT / OAuth 承担维护成本

如果后续对外部第三方开放，再考虑更细粒度的鉴权方案。

## 6. 日志与请求追踪

项目使用标准库 `logging`，没有引入 `structlog` 之类的额外依赖。

现有能力：

- 请求开始日志
- 请求结束日志
- 失败日志
- `request_id` 注入日志上下文
- 错误响应体和响应头都带 `request_id`

这是一个有意识的取舍：已经够排查问题，但不会把日志系统做得过重。

## 7. GitHub 数据设计

GitHub 数据不落库，只走 Redis TTL 缓存。

理由：

- 当前只是展示型数据，不需要长期保存
- Redis 足以降低上游请求频率
- 还能缓解匿名盗刷带来的放大效应
- 不需要为这种短生命周期数据引入数据库模型和迁移成本

读写流程：

1. 先查 Redis
2. 命中则直接返回
3. 未命中则请求 GitHub API
4. 成功后写回 Redis

## 8. 天气数据设计

天气数据使用 SQLite + SQLModel。

理由：

- 需要保留历史记录，便于查询
- 结构简单，SQLite 已足够
- SQLModel 对当前规模友好，学习和维护成本低

读写流程：

1. 先查 SQLite 最近缓存
2. 若缓存仍在 TTL 内，直接返回
3. 否则调用天气 API
4. 成功后写入 SQLite
5. 历史接口从 SQLite 读取

## 9. 健康检查设计

项目提供两类健康检查：

- `/health`：只看服务是否活着
- `/health/dependencies`：检查 Redis 和 SQLite 是否可用

这样的拆分避免把最基础的 liveness 探针与依赖检查耦合在一起。

## 10. 为什么说当前没有过度工程化

目前复杂度是可控的，原因在于：

- 只有一个 service 文件，没有过度拆分
- 只有一个数据库模型，没有引入迁移系统
- 只做 API-Key，没有过早接入复杂鉴权
- 只做 Redis 和 SQLite，不额外引入消息队列
- 文档是普通 Markdown，不引入文档站点生成链路

如果未来业务变大，建议按以下顺序逐步演进：

1. 引入更完整的测试分层
2. 增加 CI 缓存与镜像发布
3. 引入迁移工具
4. 再考虑更复杂的鉴权与权限
